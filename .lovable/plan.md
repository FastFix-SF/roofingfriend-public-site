## Goal

Make the **Portfolio page** behave like its original (pre-CRM-sync) version: the projects that are on the page today stay **exactly as they are now — frozen forever** (image, title, address, category, roof type, featured flag), and the CRM sync **only contributes brand-new projects** that aren't already in that frozen set.

The reference project `roofingfriend` was checked — its `marketing/pages/Portfolio.tsx` is the same CRM-driven implementation we already have, so there is no older static file to copy from. Instead we'll take a one-time snapshot of what the CRM is serving right now and treat that as the "originals."

## What changes

### 1. One-time snapshot script (dev-only)
Add `scripts/snapshot-portfolio.mjs` that:
- Calls the same edge function the page uses today:
  `POST https://<project>.supabase.co/functions/v1/get-portfolio` with `{ tenantId: ROOFINGFRIEND_TENANT_ID }` and the public anon key.
- Writes the response (`projects[]` + `photos{}`) verbatim to `src/data/portfolio-snapshot.json`.
- Adds a `"snapshot:portfolio"` npm script so we (and you) can re-run it manually later if you ever decide to refresh the frozen set.

This runs **once**, now, to capture the current state. After that the JSON is committed and treated as source of truth.

### 2. Frozen snapshot module
Add `src/data/portfolio-snapshot.json` (generated above) and a tiny loader `src/data/portfolioSnapshot.ts` that re-exports it as typed `ProjectRow[]` / `Record<string, PhotoRow[]>` using the existing types from `Portfolio.tsx`.

### 3. Merge rule in `src/pages/Portfolio.tsx`
Replace the current fetch-only flow with:

1. **Seed state synchronously from the snapshot** — page renders the frozen originals immediately, no loading flash for them.
2. Then call `supabase.functions.invoke("get-portfolio", …)` as today.
3. Build a `Set<string>` of snapshot project IDs.
4. From the CRM response, keep only projects whose `id` is **not** in that set → those are the "new" ones.
5. Final list = `snapshotProjects` (in their existing order) + `newCrmProjects` (sorted by `created_at` desc, as today).
6. Final photo map = `{ ...snapshotPhotos, ...newCrmPhotos }` — snapshot wins on key collisions, so frozen projects never get their photos overwritten even if someone edits them in the CRM.
7. `featured` set = featured items from the snapshot first, then any featured-and-new from CRM, capped at 3 (matches current behavior).

Filters, modal, hero section, CTA, skeletons — **untouched**.

### 4. Failure behavior
If the edge function call fails or returns empty, the page still shows the full snapshot (so the portfolio is never blank). The CRM error is silently swallowed for the user-facing list — same behavior as a fresh page with no new projects.

## Files touched

- **New** `scripts/snapshot-portfolio.mjs` — one-time / re-runnable snapshotter.
- **New** `src/data/portfolio-snapshot.json` — frozen originals (generated by the script).
- **New** `src/data/portfolioSnapshot.ts` — typed re-export of the JSON.
- **Edit** `src/pages/Portfolio.tsx` — seed from snapshot, merge CRM "new only," dedupe by `id`, snapshot wins on photos.
- **Edit** `package.json` — add `"snapshot:portfolio": "node scripts/snapshot-portfolio.mjs"`.

## Out of scope

- No edge-function changes — `get-portfolio` keeps returning everything; the merge happens client-side.
- No CRM-side changes — we are not telling the backend to stop syncing; we are just freezing the originals on this page.
- No visual / layout changes, no filter changes, no detail-modal changes.
- The Portfolio link in the navbar / footer is unchanged.
